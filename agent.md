# AGENTS.md

## 프로젝트 한줄 정의
이 프로젝트는 단체 공부 중 짧은 음성 구간을 주기적으로 분석해서,
그 구간이 공부인지 잡담인지 판정하고,
그 결과를 바탕으로 그룹의 순공 시간을 계산하는 웹 MVP다.

핵심은 "개인별 정밀 분석"이 아니라
"그룹 전체의 순공 시간과 최근 떠든 사람 / 전체 수다왕을 보여주는 것"이다.

---

## 최우선 원칙

### 1. 기술 스택 고정
반드시 아래만 사용한다.
- HTML
- CSS
- JavaScript

사용 금지:
- React
- Vue
- Svelte
- Angular
- TypeScript
- Tailwind
- 복잡한 빌드 도구
- 정식 서버 프레임워크

이 프로젝트는 순수 정적 프론트엔드 기반으로 구현한다.

---

### 2. 실제 데이터 구조
구조적으로는 API 호출처럼 설계하지만,
실제 저장과 외부 연동은 Google Apps Script + Google Spreadsheet를 사용한다.

즉,
- 프론트엔드는 직접 Spreadsheet를 다루지 않는다
- 프론트엔드는 Apps Script 웹앱 엔드포인트만 호출한다
- Apps Script가 데이터를 Spreadsheet에 저장하고 조회한다
- Apps Script가 외부 음성/STT API 호출을 중계한다

중요:
- OpenAI API 키는 프론트엔드에 넣지 않는다
- 비밀키는 Apps Script 쪽에서만 관리한다

---

### 3. 서비스 언어
사용자에게 보이는 문구는 기본적으로 모두 한국어로 작성한다.

예:
- 스터디룸 만들기
- 스터디 시작
- 최근 구간 판정
- 최근 떠든 사람
- 오늘의 수다왕
- 인정된 순공 시간

영문은 파일명, 함수명, 개발용 식별자에만 사용한다.

---

### 4. MVP 범위
이 프로젝트는 MVP다.
실제로 구현 가능한 최소 흐름만 만든다.

이번 버전에서 구현하는 핵심:
1. 스터디룸 생성
2. 참가자 등록
3. 스터디 시작
4. 짧은 음성 구간 녹음
5. 녹음 구간을 공부 / 잡담으로 판정
6. 그룹 순공 시간 계산
7. 최근 떠든 사람 표시
8. 전체 세션 기준 수다왕 계산
9. 결과 페이지 표시

이번 버전에서 하지 않는 것:
- 정밀한 실시간 음성 스트리밍
- 완전한 화자 인식
- 참가자별 음성 생체인식
- 복잡한 리포트 대시보드
- 개인별 상세 성적표 수준 분석
- 장시간 실시간 서버 인프라

## 제품의 핵심 규칙

### 그룹 순공 시간 규칙
이 서비스는 개인별 공부 시간을 따로 쌓지 않는다.
특정 구간에서 잡담이 감지되면 그 구간은 그룹 전체 순공 시간으로 인정되지 않는다.

즉:
- 공부 관련 대화면 해당 구간은 그룹 순공 시간에 추가
- 잡담이면 해당 구간은 그룹 순공 시간에 추가하지 않음

이 규칙이 UI와 데이터 처리 모두에서 가장 중요하다.

## 페이지 구성

### 1. 시작 페이지
목적:
- 스터디룸 생성
- 참가자 이름 등록
- 스터디 세션 시작

필수 UI:
- 스터디룸 이름 입력
- 참가자 이름 추가/삭제
- 시작 버튼

주의:
- 참가자별 목소리 등록은 이번 MVP에서 필수 구현이 아니다
- 나중에 붙일 수 있도록 데이터 구조만 열어둔다

---

### 2. 스터디 페이지
목적:
- 현재 스터디 상태를 교실 장면 중심으로 보여준다
- 텍스트 대시보드가 아니라 "교실 + 큰 칠판"이 중심이다

핵심 UI 원칙:
- 화면 중심에는 교실 장면이 있다
- 중앙에 큰 칠판이 있다
- 핵심 정보는 칠판 위에 적는다
- 긴 로그/긴 설명/세부 표는 최대한 줄인다

칠판에 보여줄 정보:
- 전체 스터디 시간
- 인정된 순공 시간
- 최근 구간 판정 결과
- 최근 녹음에서 떠든 사람
- 현재까지 가장 많이 떠든 사람

예시 문구:
- 전체 스터디 시간: 35분
- 인정된 순공 시간: 20분
- 최근 구간: 잡담 감지
- 최근 떠든 사람: 민수
- 오늘의 수다왕: 지현

중요:
- 개인별 상세 분석 텍스트를 길게 뿌리지 않는다
- 긴 카드 목록이나 긴 로그 목록을 메인으로 쓰지 않는다
- "교실 장면 + 칠판 요약"이 최우선이다

---

### 3. 결과 페이지
목적:
- 스터디 종료 후 그룹 결과 요약 표시

필수 표시 정보:
- 총 스터디 시간
- 인정된 순공 시간
- 제외된 시간
- 가장 많이 떠든 사람
- 전체 구간 수
- 공부로 인정된 구간 수
- 잡담으로 제외된 구간 수

표현 방식:
- 칠판 또는 교실 게시판 같은 느낌으로 요약
- 복잡한 차트보다 간단한 텍스트 요약 우선

## 실제 구현 흐름

### 전체 흐름
1. 사용자가 시작 페이지에서 스터디룸과 참가자를 만든다
2. 스터디 시작 버튼을 누르면 세션이 생성된다
3. 사용자는 스터디 페이지에서 녹음을 시작한다
4. 일정 주기마다 짧은 음성 구간을 만든다
5. 프론트엔드는 해당 음성 파일을 Apps Script로 전송한다
6. Apps Script는 외부 음성/STT API로 분석을 요청한다
7. 분석 결과를 바탕으로 해당 구간을 study/chat으로 판정한다
8. Apps Script는 결과를 Spreadsheet에 저장한다
9. 프론트엔드는 최신 요약 데이터를 불러와 칠판 UI를 갱신한다
10. 사용자가 종료하면 결과 페이지에서 최종 요약을 보여준다

## 음성 분석 정책

### 현실적인 MVP 처리 원칙
음성 분석은 "완벽한 화자 인식"이 아니라,
"짧은 음성 구간을 텍스트화하고, 구간 전체를 공부인지 잡담인지 판정하는 것"이 중심이다.

### 판정 우선순위
1. 공부 / 잡담 판정
2. 최근 떠든 사람 추정
3. 누적 발화량 집계

### 최근 떠든 사람 처리
이번 MVP에서는 다음 중 하나를 허용한다.
- diarization 결과가 있으면 가장 많이 발화한 화자를 최근 떠든 사람으로 사용
- diarization이 불안정하면 최근 구간의 "주요 화자"만 표시
- 최악의 경우 mock 또는 단순 추정으로 대체 가능

중요:
"최근 떠든 사람"은 MVP에서 완전한 실명 식별 정확도를 보장하지 않아도 된다.
하지만 "그룹 순공 판정"은 반드시 동작해야 한다.

## 디자인 규칙

### 전체 분위기
- 장난스러운 학교/교실 분위기
- 큰 칠판 중심
- 일러스트형 화면
- 귀엽고 직관적인 연출
- 진지한 SaaS 대시보드 스타일 금지

### 피해야 할 것
- 긴 텍스트 로그를 메인에 놓는 것
- 세부 통계를 카드로 잔뜩 나열하는 것
- 기업용 관리자 페이지 같은 UI
- 지나치게 복잡한 정보 구조

### 꼭 유지할 것
- 교실 장면이 먼저 보여야 한다
- 칠판이 정보의 중심이어야 한다
- 사용자는 한눈에 그룹 상태를 이해할 수 있어야 한다
